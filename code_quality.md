# Code Quality Metrics

Designing code that is maintainable is a struggle.  Many projects have been doomed by poor design choices, or worse - continued to limp along for years or decades when it should have been abandoned in favor of a rewrite.  Often people don't realize the cost of **technical debt** - choices made early on (often for the sake of expediency or to save a tiny bit of money or time now), that cost us enormous amounts of time and money later on.

As AI takes over more of the low-level programming tasks, humans will undoubtedly trend to the design-side of things.  This means we need to be able to vet code given to us by an AI (or junior programmer) and determine if it is of sufficient quality to merge into our code base.  To aid us in that endeavor are code quality metrics.

## Metrics

No single metric can tell you if your code is "good enough".  But there are many facets of our code that we can measure, and sometimes an increase in one metric can cause a decrease in another.  As an architect of a system, you must determine what you or your organization needs.  In grad school, I had a professor who had worked for Boeing.  She used to constantly remind us that it is completely possible to build an airplane that is essentially crash-proof.  The reason we don't is that it would cost too much to do so.  As sad a statement as that is, it is largely true - given a limited set of resources we must optimize to get the most good out of them.  And we will always lack resources.

While there are a great many metrics available, we will be learning about the following:

- Cyclomatic Complexity
- Test Coverage
- Code Duplication
- Code Churn
- Technical Debt
- Maintainability Index
- Security
- Bug Rate
- Standards Compliance
- Documentation Quality

### Cyclomatic Complexity

Imagine you have a function called `my_function`.  It does something really important and novel.  In your function, you have an `if`-statement:

```python
def my_function(self):
  if some_check():
    ... run this code ...
  else:
    ... run this code instead ...
```

Looking closely at this pseudocode, we can see that there are two paths of execution through the function - either `some_check()` returns `True` and the first code block runs, or it returns `False` and the second code block runs.  This is an example of **cyclomatic complexity** - the number of independent paths through a function.  Now, imagine that a few lines below this code there is another `if`-statement check - now there are four paths through the code.  The possible paths are:

| First Path Result | Second Path Result |
|-----------------|------------------|
| `True`          | `True`           |
| `True`          | `False`          |
| `False`         | `True`           |
| `False`         | `False`          |

For every additional branch we increase the complexity - sometimes exponentially.  Complex code is hard to maintain.  As we've already talked about in this book, humans aren't great at long, complicated processes.  We just don't think that way.  As cyclomatic complexity increases, so do the number of bugs.

Calculating this type of complexity usually involves a **node-edge graph** called a **flow-graph**:

:::{figure} ./images/flow_graph.png
:width: 100%
:align: center

Figure 1: Sample flow graph with two branches.
:::

If we view each of the arrows in the diagram as edges of a graph ($E$), and the other shapes as nodes ($N$), we can use the formula

$$M = E - N + 2$$

to calculate the cyclomatic complexity.  For our simple example here, that would be 

$$M = 10 - 9 + 2$$
$$M = 3$$

A value less than $10$ is generally considered a simple function with low risk.  However, with all metrics, the number needs to be considered in the context of other factors and along with other metrics.

To calculate the cyclomatic complexity (and a few other metrics), we can use the `radon` library ([Radon Documentation](https://radon.readthedocs.io/en/latest/)).  Consider this function written for a text-based adventure game (fair warning - it is long!):

```python
 def process_player_input(command, player, world):
 '''Generated by ChatGPT 5.5.'''
    command = command.lower().strip()

    if command == "":
        return "You must enter a command."

    words = command.split()

    if len(words) == 1:
        verb = words[0]
        target = None
    else:
        verb = words[0]
        target = " ".join(words[1:])

    # Movement commands
    if verb == "go" or verb == "move" or verb == "walk":
        if target is None:
            return "Go where?"

        if target == "north":
            if "north" in world[player["location"]]["exits"]:
                player["location"] = world[player["location"]]["exits"]["north"]

                if player["location"] == "dark_cave":
                    if "torch" not in player["inventory"]:
                        player["health"] -= 10
                        return "You stumble in the dark cave and lose 10 health."

                return f"You move north to {player['location']}."
            else:
                return "You cannot go north."

        elif target == "south":
            if "south" in world[player["location"]]["exits"]:
                player["location"] = world[player["location"]]["exits"]["south"]
                return f"You move south to {player['location']}."
            else:
                return "You cannot go south."

        elif target == "east":
            if "east" in world[player["location"]]["exits"]:
                if world[player["location"]].get("locked_east"):
                    if "rusty key" in player["inventory"]:
                        world[player["location"]]["locked_east"] = False
                        player["location"] = world[player["location"]]["exits"]["east"]
                        return "You unlock the eastern gate and proceed."
                    else:
                        return "The eastern gate is locked."
                else:
                    player["location"] = world[player["location"]]["exits"]["east"]
                    return f"You move east to {player['location']}."
            else:
                return "You cannot go east."

        elif target == "west":
            if "west" in world[player["location"]]["exits"]:
                player["location"] = world[player["location"]]["exits"]["west"]
                return f"You move west to {player['location']}."
            else:
                return "You cannot go west."

        else:
            return "Unknown direction."

    # Inventory commands
    elif verb == "take" or verb == "grab" or verb == "pick":
        if target is None:
            return "Take what?"

        room_items = world[player["location"]]["items"]

        if target in room_items:
            if len(player["inventory"]) >= player["max_inventory"]:
                return "Your inventory is full."

            if target == "cursed idol":
                player["cursed"] = True
                player["health"] -= 20

            player["inventory"].append(target)
            room_items.remove(target)

            if target == "gold coin":
                player["gold"] += 1

            return f"You take the {target}."
        else:
            return f"There is no {target} here."

    elif verb == "drop":
        if target is None:
            return "Drop what?"

        if target in player["inventory"]:
            player["inventory"].remove(target)
            world[player["location"]]["items"].append(target)

            if target == "torch":
                if player["location"] == "dark_cave":
                    player["health"] -= 5
                    return "You dropped the torch. Darkness surrounds you."

            return f"You dropped the {target}."
        else:
            return "You do not have that item."

    elif verb == "inventory" or verb == "inv":
        if len(player["inventory"]) == 0:
            return "Your inventory is empty."
        else:
            return "Inventory: " + ", ".join(player["inventory"])

    # Combat commands
    elif verb == "attack" or verb == "fight":
        if target is None:
            return "Attack what?"

                damage = 25
            elif "dagger" in player["inventory"]:
                damage = 10
            else:
                damage = 2

            if target == "dragon":
                if "magic shield" not in player["inventory"]:
                    player["health"] -= 50

                    if player["health"] <= 0:
                        return "The dragon burns you alive."

                if "dragon" in enemies:
                    enemies.remove("dragon")
                    player["xp"] += 100
                    return "You slay the dragon!"

            elif target == "goblin":
                player["health"] -= 5

                if player["health"] <= 0:
                    return "The goblin defeats you."

                enemies.remove("goblin")
                player["xp"] += 10
                return "You defeat the goblin."

            elif target == "ghost":
                if "holy water" in player["inventory"]:
                    enemies.remove("ghost")
                    player["xp"] += 30
                    return "The ghost vanishes."
                else:
                    return "Your attacks pass through the ghost."

            else:
                enemies.remove(target)
                player["xp"] += 5
                return f"You defeat the {target}."

        else:
            return "There is nothing like that here."

    # Use item commands
    elif verb == "use":
        if target is None:
            return "Use what?"

        if target not in player["inventory"]:
            return "You do not have that item."

        if target == "health potion":
            if player["health"] >= 100:
                return "Your health is already full."

            player["health"] += 30

            if player["health"] > 100:
                player["health"] = 100

            player["inventory"].remove("health potion")
            return "You drink the health potion."

        elif target == "torch":
            if player["location"] == "dark_cave":
                return "The cave is illuminated."

            return "You wave the torch around."

        elif target == "magic scroll":
            if player["mana"] >= 50:
                return "Your mana is already full."

            player["mana"] += 50

            if player["mana"] > 50:
                player["mana"] = 50

            player["inventory"].remove("magic scroll")
            return "Arcane energy fills your body."

        elif target == "rusty key":
            if world[player["location"]].get("locked_chest"):
                world[player["location"]]["locked_chest"] = False
                return "You unlock the chest."
            else:
                return "Nothing happens."

        else:
            return "You cannot use that item."

    # Dialogue commands
    elif verb == "talk":
        if target is None:
            return "Talk to whom?"

        npcs = world[player["location"]].get("npcs", [])

        if target in npcs:
            if target == "merchant":
                if player["gold"] >= 10:
                    return "Merchant: Interested in rare goods?"
                else:
                    return "Merchant: Come back when you have more gold."

            elif target == "wizard":
                if player["xp"] >= 100:
                    player["spells"].append("fireball")
                    return "Wizard: I shall teach you fireball."
                else:
                    return "Wizard: You are not experienced enough."

            elif target == "guard":
                if player.get("criminal_record"):
                    return "Guard: You're under arrest!"
                else:
                    return "Guard: Stay out of trouble."

            else:
                return f"You speak with {target}."

        else:
            return "Nobody by that name is here."

    # Look command
    elif verb == "look":
        room = world[player["location"]]

        description = room["description"]

        if len(room["items"]) > 0:
            description += "\nItems: " + ", ".join(room["items"])

        if len(room.get("enemies", [])) > 0:
            description += "\nEnemies: " + ", ".join(room["enemies"])

        if len(room.get("npcs", [])) > 0:
            description += "\nPeople: " + ", ".join(room["npcs"])

        description += "\nExits: " + ", ".join(room["exits"].keys())

        return description

    # Status command
    elif verb == "status":
        return (
            f"Health: {player['health']}\n"
            f"Mana: {player['mana']}\n"
            f"XP: {player['xp']}\n"
            f"Gold: {player['gold']}"
        )

    # Quit command
    elif verb == "quit":
        player["quit"] = True
        return "Game over."

    else:
        return "Unknown command."
```

### Test Coverage

I started a job many years ago as a systems programmer for a small department that was part of a larger organization.  A few weeks in, I had started coding a project when my boss called.  He asked how the project was going, and I told him, "Great!  I just need to finish writing test cases."  He replied, "We don't do that here, just hack it together and make it work."  I was a bit surprised, as I was taught to always test your code.  A few years later (after my boss had retired), our department was merged into the larger organization's I.T. infrastructure.  Almost immediately, code my boss had written began to fail as we tried to integrate it into the larger ecosystem.  Bugs and security vulnerabilities were rampant in his code (in his defense his training was in business, not computer science).  Ultimately, almost everything he had written had to be scrapped.

The truth is that code that hasn't been tested should be considered broken.  It is also true that it is usually impossible to fully test our code.  There are just too many paths of execution through a program for us to test everything completely.  However, that doesn't mean we shouldn't try to test as much as we can.  **Test Coverage** metrics help us to understand which parts of our code are being tested.

There are different levels of coverage, as we discussed in the "Testing Code" chapter. 

### Code Duplication

Every programmer has, at some point, copied a chunk of code, pasted it somewhere else, and changed a variable name or two.  It feels efficient in the moment - why rewrite something that already works?  The problem shows up later, when a bug is discovered in that pasted logic and has to be fixed in five different places instead of one.  Miss even one of those copies, and you've just shipped a bug that "was already fixed."

**Code duplication** (sometimes called "copy-paste programming") measures how much of your codebase is made up of repeated blocks of logic rather than shared, reusable pieces.  It's usually reported as a percentage:

$$\text{Duplication \%} = \frac{\text{Duplicated Lines}}{\text{Total Lines}} \times 100$$

A small amount of duplication is unavoidable and often harmless - two functions that happen to both check if a string is empty aren't a crisis.  But when whole blocks of business logic are duplicated across a codebase, it's usually a sign that an abstraction is missing: a shared function, a base class, a utility module.  This violates one of the oldest principles in software engineering, called the **DRY** principle - meaning "Don't Repeat Yourself."

Tools like `jscpd` (JavaScript, and many other languages), PMD's Copy-Paste Detector (Java), and `flake8-duplicate-code` or `pylint`'s duplicate-code checker (Python) scan your source for near-identical blocks and report where they occur.  In our adventure-game example above, notice how the pattern `if target is None: return "___ what?"` appears at the top of nearly every verb branch.  That's a small, low-risk duplication - but at scale, across a whole codebase, these little repeated patterns add up to a maintenance headache.

### Code Churn

**Code churn** measures how frequently a piece of code changes over time - how many lines are added, modified, or deleted in a file or function across commits.  Unlike most of the other metrics in this chapter, churn isn't calculated from a single snapshot of your code; it's calculated from your project's *history*, usually pulled straight out of your version control system.

A simple way to think about churn is:

$$\text{Churn} = \text{Lines Added} + \text{Lines Deleted} + \text{Lines Modified}$$

measured over some window of time, per file or per function.

Why does this matter?  Churn is one of the best predictors we have of where bugs are hiding.  Files that change constantly are either evolving features (expected) or being repeatedly patched because something about them just won't stay fixed (a red flag).  A file with high churn *and* high cyclomatic complexity is a particularly dangerous combination - it's complicated code that people keep having to touch, which is exactly the kind of code where a small change has unintended side effects.

You don't need a fancy tool to get a rough sense of churn - `git` will tell you:

```bash
git log --since="6 months ago" --numstat -- process_command.py | \
  awk '{added+=$1; deleted+=$2} END {print "Added:", added, "Deleted:", deleted}'
```

If you see a file near the top of your churn list every sprint, it's worth asking: is this file poorly designed, or is it just a natural hotspot of business activity?  The metric doesn't answer that question for you - but it tells you exactly where to go looking.

### Technical Debt

We already introduced the term **technical debt** at the start of this chapter, but it's worth sitting with the metaphor for a moment, because it was chosen deliberately.  The programmer Ward Cunningham coined the term to describe the shortcuts we take to ship something quickly, with the understanding that we'll pay down the "principal" later by cleaning it up.  Like financial debt, technical debt isn't inherently bad - sometimes taking on debt is the right call, the same way a small business might take out a loan to open its doors sooner.  The danger is *interest*.  Debt that isn't repaid keeps accruing, and unpaid technical debt shows up as bugs, slow feature development, and engineers who are afraid to touch certain files.

It's common to categorize technical debt along two axes: whether it was taken on *deliberately* or *inadvertently*, and whether it was *reckless* or *prudent*:

| | Deliberate | Inadvertent |
|---|---|---|
| **Reckless** | "We don't have time for tests." | "What's a design pattern?" |
| **Prudent** | "We'll ship now and refactor after launch." | "Now we know how we should have built it." |

Measuring technical debt directly is difficult, since it's more of a concept than a raw count.  In practice, tools like SonarQube estimate it as a **technical debt ratio** - essentially, an estimate of how many hours it would take to fix every code smell, complexity hotspot, and duplication issue in the codebase, divided by the estimated hours it took to build it in the first place:

$$\text{Technical Debt Ratio} = \frac{\text{Remediation Cost}}{\text{Development Cost}} \times 100$$

A ratio under 5% is generally considered healthy; above 20% or so, a project is often heading toward the "should we rewrite this" conversation mentioned at the start of this chapter.

### Maintainability Index

If cyclomatic complexity, code duplication, and lines of code are each individual ingredients, the **Maintainability Index (MI)** is an attempt to bake them into a single number that summarizes "how easy is this code to maintain?"  It was originally proposed by researchers at the University of Idaho and Hewlett-Packard in the 1990s, and it remains one of the most widely used composite metrics today.

The classic formula is:

$$MI = 171 - 5.2\ln(V) - 0.23(G) - 16.2\ln(LOC)$$

where $V$ is the Halstead Volume (a measure of how much information is packed into the code, based on the number of operators and operands), $G$ is the cyclomatic complexity, and $LOC$ is the number of lines of code.  The result is typically rescaled to fall between 0 and 100, where higher is better:

- **85–100**: highly maintainable
- **65–84**: moderately maintainable
- **Below 65**: difficult to maintain

Like cyclomatic complexity, `radon` can calculate this for us directly:

```python
radon mi code_samples/process_command.py -s
```

which would output something like:

```python
code_samples/process_command.py - C (34.87)
```

A score of 34.87 confirms what we already suspected from the cyclomatic complexity score alone - this file is difficult to maintain, and any change to it carries real risk.  The value of MI is that it lets you triage an entire codebase quickly: sort every file by MI, and you instantly know where to focus your refactoring effort first.

### Security

Not every code quality problem is about readability or maintainability - some are about whether your code can be exploited.  **Security metrics** attempt to quantify how vulnerable a codebase is to attack, and they're often the metrics that matter most to the business, since a security breach can cost far more than a hard-to-read function ever will.

Static Application Security Testing (SAST) tools scan source code for known-dangerous patterns without ever running the program - things like SQL queries built by string concatenation (an opening for SQL injection), hard-coded credentials, use of weak or deprecated cryptographic functions, or unsanitized user input flowing into a shell command.  For Python, a common tool is `bandit`:

```bash
bandit -r code_samples/
```

which reports each finding along with a severity (low, medium, high) and confidence level.  Other useful measures in this space include:

- **Number of known vulnerabilities**, often cross-referenced against the CVE (Common Vulnerabilities and Exposures) database for third-party dependencies
- **OWASP Top 10 coverage** - whether the codebase has been checked against the most common categories of web application vulnerability
- **Mean time to patch** - how quickly a known vulnerability is fixed once discovered

Security metrics are unusual among the metrics in this chapter because a "perfect" score doesn't really exist - new vulnerability classes are discovered constantly, and yesterday's secure code can become tomorrow's cautionary tale.  Security has to be treated as an ongoing process, not a one-time checkbox.

### Bug Rate

**Bug rate** (also called **defect density**) is one of the more intuitive metrics in this chapter: it simply measures how many bugs show up in a piece of code, normalized by its size so that a 10,000-line file isn't unfairly compared to a 100-line one:

$$\text{Defect Density} = \frac{\text{Number of Defects}}{\text{Thousand Lines of Code (KLOC)}}$$

Bug rate is typically tracked over the lifetime of a release, pulling data from an issue tracker (bugs filed and tagged against a given module) and correlating it with the size of that module.  It's a *lagging* indicator - unlike cyclomatic complexity or the maintainability index, which try to predict where problems will occur, bug rate tells you where problems have already occurred.  That makes it extremely useful for validating whether your other metrics are actually meaningful for your project: if the modules with the worst cyclomatic complexity and maintainability index scores also have the highest bug rates, you've got good evidence those metrics are worth paying attention to going forward.

It's worth noting that a low bug rate doesn't automatically mean good code - it might just mean nobody has found the bugs yet, or that the module barely gets used.  As with every metric in this chapter, context matters.

### Standards Compliance

Every team, language community, or organization tends to settle on a set of conventions for how code should look: naming conventions, indentation, import ordering, maximum line length, and so on.  **Standards compliance** measures how consistently a codebase follows these agreed-upon rules.

This might sound like a purely cosmetic concern, but consistency has real value.  When every file in a codebase follows the same conventions, an engineer moving from one part of the system to another doesn't have to spend mental energy figuring out a new set of stylistic quirks - they can focus entirely on the logic.  Code review also moves faster, since reviewers aren't distracted arguing about tabs versus spaces or where a brace should go.

For Python, standards compliance is usually measured against **PEP 8**, the language's official style guide, using a linter such as `flake8` or `pylint`:

```bash
flake8 code_samples/process_command.py
```

which reports each violation along with a line number and rule code, for example:

```python
code_samples/process_command.py:12:80: E501 line too long (91 > 79 characters)
```

Most teams track compliance as a simple pass/fail gate in continuous integration - a pull request can't merge until the linter reports zero violations - rather than as a graded percentage, since style rules are (mostly) binary: either you followed them or you didn't.

### Documentation Quality

The last metric in our tour is one that's frequently overlooked: **documentation quality**.  Code that works perfectly but can't be understood by the next person who has to touch it is still a liability.  A function's docstring, a module's README, or a system's architecture diagram are all part of the codebase in a very real sense, even though they never execute.

Documentation quality can be measured in a few different ways:

- **Docstring coverage** - the percentage of functions, classes, and modules that have an accompanying docstring.  Tools like `interrogate` (Python) can scan a project and report this as a single number, much like test coverage tools do for tests.
- **Comment density** - the ratio of comment lines to code lines.  Note that this metric has to be interpreted carefully; a high comment density can indicate thorough documentation, or it can indicate code so confusing that every other line needed an explanation.  More comments are not automatically better.
- **Freshness** - whether documentation has been updated alongside the code it describes.  Stale documentation that describes behavior the code no longer has is often worse than no documentation at all, since it actively misleads the reader.

Running `interrogate` against our adventure-game example makes the problem obvious:

```bash
interrogate code_samples/process_command.py -v
```

```python
code_samples/process_command.py: 0.0% docstring coverage (0/1)
```

A single one-line docstring (`'''Generated by ChatGPT 5.5.'''`) tells us who - or what - wrote the function, but says nothing about what the function actually does, what arguments it expects, or what it returns.  For a function this large and this complex, that's a significant gap - anyone inheriting this code has to reverse-engineer its behavior line by line before they can safely change it.

## Putting It All Together

None of these metrics should be used in isolation, and none of them should be treated as a strict pass/fail gate on their own.  A function can have low cyclomatic complexity and still be full of bugs.  A file can have 100% test coverage and still be nearly impossible to read.  The real value of code quality metrics is in the *pattern* they reveal when looked at together: a file with high complexity, high churn, low maintainability index, and a high bug rate is telling you something very different than a file that scores poorly on just one of those measures.

As more of our code is written by AI assistants rather than typed by hand, these metrics become even more important, not less.  It has never been easier to generate code that *looks* plausible.  Code quality metrics are one of the best tools we have for verifying that "looks plausible" and "is actually good" are the same thing.
